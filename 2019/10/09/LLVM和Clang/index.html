LLVM & Clang

LLVM Clang
发布日期:   2019-10-09

语言类型

  • 编译型语言
    • C++ OC Swift
    • 通过编译生成机器码，机器码在CPU上运行
    • 执行效率高
    • 调试周期长
  • 直译型（脚本）语言
    • JS Python
    • 不需要编译，通过解释器将代码解释为CPU可以执行的代码
    • 编写调试方便
    • 执行效率低

图解区分：

|-----------| |--------| |-------|     |-------| 
|Source Code|-|Compiler|-|Macho-o|---> |Macho-O|//for OC 
|-----------| |--------| |-------|     |-------| 

|----------|    |-----------| |-----------|
Source Code|--->|Source Code|-|Interpreter|//for Python perl shell
|----------|    |-----------| |-----------|

从上图可知：

  • 编译型语言需要预编译AOT(ahead of time)
  • 直译型语言可以即时编译JIT（just in time）

iOS编译器

  • Xcode5之前，使用GCC
  • Xcode5开始，使用LLVM

LLVM包括编译器前端，优化器和编译器后端三个模块。OC和Swift的区分在于前端编译器。

|Clang|----->LLVM Optimizer------>LLVM Code Generator//for OC

LLVM

LLVM IR

IR是intermediate representation的缩写，它是整个编译过程中生成的区别于源码和机器码的中间代码。
IR提供了独立于任何机器架构的源语，因此它是LLVM优化和进行代码生成的关键，也是LLVM有别于全其他编译器的最大特点。

为什么需要IR？
编译器的架构分为，前端，优化器和后端。GCC的前端和后端没有完全分离，耦合在一起，因而如果要支持一门新的语言或者硬件
平台，需要做大量的工作。而LLVM和传统编译器的最大不同在于，前端输入任何语言，经过编译器前端处理后，生成的中间码都是IR格式的。这样的优点是，如果需要支持一个新的语言，我们只需要实现一个新的前端，如果我们需要支持一个新的硬件设备，我们只需要实现一个新的后端。

Bitcode

Bitcode就是IR的一种，存储在磁盘上的二进制文件（.bc结尾）
Apple会单独对Bitcode进行额外的优化。
从Xcode7开始，苹果支持在提交App编译产物的同时提交Bitcode，并且之后对提交的Bitcode的App都单独进行了云端编译打包。也就是说吗，即使在提交时已经将本地编译好的ipa提交到Appstore，苹果最终还是会使用Bitcode在云端打包，用户最终下载到手机上的版本就是云端编译处理的版本，而非开发人员本地编译的版本。

为什么要Bicode

  • 可以做一些额外的优化工作
  • 减少安装包的大小
  • 由于Bicode是无关设备架构的，它可以被转为任何被支持的CPU架构，包含还没有发明的CPU架构。如果苹果以后推出一个新架构和手机，苹果可以直接进行编译，无需开发人员重新编译上传。

Clang 和 Swift

Clang是LLVM的子项目，是OC的编译器，目的是替代GCC，Clang在OC编译过程中，扮演了编译器前端的角色，同时也参与Swift编译过程中的OC API映射阶段。

Clang相对于GCC，速度大概是GCC的3倍，生成AST(语法树)所有的内存是GCC的五分之一。

Clang的主要工作：

  • 预处理，比如宏展开，头文件导入，去除注释；
  • 词法分析，切割代码，比如大小括号，等于号，字符串等
  • 语法分析，验证语法是否正确
  • 生成AST，将所有节点组成抽象语法数AST
  • 静态分析，分析代码是否存在问题，给出错误信息和修复方案
  • 生成LLVM IR，将语法树自顶向下遍历逐步翻译成LLVM IR
